# Incident Report: 2026-03-11 - Backend CI/CD Failures

## Summary

On 2026-03-11, both `cv-backend` and `cv-admin-backend` experienced multiple CI/CD failures across tests, Docker builds, and smoke tests. These issues were caused by a combination of TypeScript configuration regressions, legacy script dependencies, and environment synchronization gaps.

## Timeline (UTC)

- **18:00**: Initial failures detected in `cv-backend` Docker build (`bun run build` failing with module resolution errors).
- **19:30**: `cv-admin-backend` unit tests found to be unstable and using legacy shell-based test triggers.
- **21:00**: Unified smoke testing strategy proposed to replace fragmented scripts.
- **22:15**: `cv-env` updated with centralized `docker-smoke-test` action.
- **23:30**: `cv-backend` build fixed by adjusting `tsconfig.build.json`.
- **00:15 (Next Day)**: `cv-admin-backend` smoke test fails due to missing `buns` command in production image.
- **00:30**: Final fix pushed for `admin-backend` (switching `buns` to `bun`).
- **00:45**: Both CI pipelines confirmed green.

## Root Cause Analysis

### 1. TypeScript Module Resolution (Backend)

**Issue**: `tsc` was unable to locate the generated Prisma client during the Docker build stage.
**Cause**: The `moduleResolution` setting in `tsconfig.build.json` was set to a value incompatible with the generated Prisma output in the container environment.
**Fix**: Changed `moduleResolution` to `Node` to ensure standard path lookup.

### 2. Legacy Test Infrastructure (Admin Backend)

**Issue**: Admin backend used complex, error-prone shell scripts (`services-test-start.sh`) for API tests.
**Cause**: Fragmentation of testing strategies across repositories.
**Fix**: Simplified to a "one-shot" `docker compose run --profile test test-runner` pattern, unifying it with the main backend.

### 3. Runtime Command Mismatch (Admin Backend)

**Issue**: Production container crashed on startup.
**Cause**: The `package.json` scripts used `buns` (a local development wrapper) instead of the standard `bun` command available in the Docker environment.
**Fix**: Updated `start` and `dev` scripts to use `bun`.

### 4. Smoke Test Fragmentation

**Issue**: Smoke tests had different logic, custom log tailing, and different regex for error assertions across repos.
**Cause**: Organic growth of testing scripts without central coordination.
**Fix**: Centralized logic into `cv-env` with a shared script that handles log streaming, multi-line exclusions, and exit code management.

## Steps Taken

1. **Centralized Smoke Testing**: Created unified action in `cv-env`.
2. **Synchronized Environments**: Updated `flake.nix/lock` across repos to use latest `cv-env` (including SOPS discovery fixes).
3. **Upgraded Actions**: Moved all workflows to Node 24 compatible Actions (v5/v6).
4. **Hardened Docling Integration**: Added healthchecks to prevent race conditions during smoke tests.
5. **Fixed Build Artifacts**: Corrected `tsconfig` and updated `bun.lock` to fix frozen lockfile errors in CI.

## Prevention

- **Standardization**: All new backends must use the `test-runner` Docker Compose pattern.
- **Linting**: CI now enforces log hygiene; `ERROR` or `WARN` logs without explicit exclusions will fail the build.
- **Dependency Management**: Use `bun run check` locally before pushing to catch lockfile or type-check regressions early.
